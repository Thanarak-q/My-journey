## สิ่งที่ได้เรียนรู้คือ

Web caching ถูกออกแบบมาเพื่อลดจำนวนรีเควสต์ที่ต้องส่งไปยัง backend และช่วยให้เว็บตอบสนองได้เร็วขึ้น แต่ถ้าระบบมีการ serialize หรือ normalize path/URL ไม่ดีพอ ก็อาจเปิดช่องให้ threat actor อาศัยความต่างในการตีความ URL ระหว่าง cache กับ origin server มาใช้โจมตีได้

ใน lab ของ PortSwigger มีสถานการณ์ประมาณว่า เรามี endpoint /my-account ซึ่งปกติ Nginx จะไม่ cache หน้าแบบนี้ และจะ cache เฉพาะ path ที่มองว่าเป็น static resource เท่านั้น เช่น path ใต้ /resources หรือไฟล์บางนามสกุลอย่าง .js

ดังนั้นสิ่งที่ต้องทำคือ

ทำให้ origin server เชื่อว่ารีเควสต์นี้กำลังเข้าถึง /my-account
ทำให้ cache/Nginx เชื่อว่ารีเควสต์เดียวกันนี้เป็น static resource ที่ cache ได้
เมื่อ victim เปิด URL นี้ก่อน cache ก็จะเก็บ response ของ victim ไว้
หลังจากนั้น threat actor ก็สามารถเรียก URL เดิมซ้ำ เพื่อดึง cached response ของ victim ออกมาได้

ระหว่างทำ lab เราจึงต้องค่อย ๆ ทดสอบว่า origin กับ cache ตีความ path เหมือนกันหรือไม่ เช่น การเติม suffix อย่าง abc.js เข้าไป ซึ่งมีไว้เพื่อเช็กว่า cache จะมองรีเควสต์นี้เป็นไฟล์ static หรือเป็น resource ที่ cache ได้หรือไม่ และในบางกรณีก็ช่วยให้ attacker มี URL ที่ชัดเจนสำหรับเรียก cached response ของ victim กลับมาดูได้ง่ายขึ้นด้วย อีกส่วนหนึ่งก็คือการลอง path traversal หรือ dot-segment อย่าง ..%2f เพื่อดูว่าใครเป็นฝ่าย normalize path ก่อน

ถ้า origin กับ cache ตีความ URL ไม่เหมือนกันเมื่อไร ก็จะเกิดช่องให้หลอก origin ว่ารีเควสต์นี้คือ /my-account แต่หลอก cache ว่าเป็น resource ที่ cache ได้ และนั่นก็คือจุดที่ทำให้ข้อมูลของ victim ถูกเก็บไว้ใน cache และถูกดึงออกมาได้ภายหลั
